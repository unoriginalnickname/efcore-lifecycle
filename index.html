🔧 Design-Time — runs once at development / deployment
① DbContext
📦 DbContext — gateway between your code and the database class: design-time · instance: per-scope
DbContext is two things at once. The class you define holds configuration, DbSet<T> properties, and Fluent API mappings — written once, compiled once on first instantiation. The instance resolved per scope holds the Change Tracker, the open connection, and the in-flight Unit of Work. These lifetimes are completely different. Confusing them — especially registering as Singleton — is the source of most DbContext bugs.
1
Inherit DbContext and expose DbSet<T> properties — one per entity aggregate root
EF uses DbSet<T> as the LINQ entry point for queries and the anchor for change tracking. You don't need a DbSet for every entity — related entities reachable via navigation properties (e.g. Post.Comments) are discovered automatically. Only aggregate roots need their own set.
2
Override OnModelCreating to apply Fluent API configuration, or leave it empty and rely on conventions and annotations
This is where the model is described. EF reads it on first instantiation — in web apps this is typically the first request — compiles it into an immutable cache, and reuses that cache for every subsequent instance. See ② Model for the full pipeline.
3
Register with AddDbContext<T> in Program.cs, specifying provider and connection string
Always Scoped (the default) — one instance per scope, disposed when the scope ends. Singleton silently corrupts data by sharing the Change Tracker across concurrent requests. Transient wastes connection pool slots and breaks batching across services. For non-web tools, CLIs, or tests, use OnConfiguring instead — it configures the context directly without a DI container.
// ── App path — DI-registered, Scoped (web apps, APIs) ──────────────────────
public class AppDbContext : DbContext
{
    // Aggregate roots only — comment entities reached via Post.Comments need no DbSet
    public DbSet<User> Users { get; set; } = null!;
    public DbSet<Post> Posts { get; set; } = null!;

    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    protected override void OnModelCreating(ModelBuilder mb)
    {
        // Fluent API configuration — see ② Model
        // Called on first instantiation; result cached for app lifetime
    }

    // ── Tool / CLI / test path — OnConfiguring, no DI container needed ─────
    // IsConfigured guard ensures this is skipped when AddDbContext already ran
    protected override void OnConfiguring(DbContextOptionsBuilder ob)
    {
        if (!ob.IsConfigured)
        {
            // Read from environment variable — never hardcode a connection string
            // Double underscore (__) is the ASP.NET Core convention for nested config keys
            // e.g. Azure App Service → ConnectionStrings__Default maps to ConnectionStrings:Default
            var cs = Environment.GetEnvironmentVariable("ConnectionStrings__Default")
                     ?? throw new InvalidOperationException("Connection string not set");
            ob.UseSqlServer(cs);
        }
    }
}

// In Program.cs — before builder.Build():
builder.Services.AddDbContext<AppDbContext>(o =>
    o.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// For MVC projects, also register controllers and views:
builder.Services.AddControllersWithViews();   // registers controllers, views, and Razor

var app = builder.Build();

// … UseStaticFiles, UseRouting, UseAuthentication, UseAuthorization …

app.MapControllerRoute(                        // wire up conventional MVC routing
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");
💡 MVC projects need AddControllersWithViews() — it's not included in AddDbContext and easy to miss when setting up EF Core outside a template. Without it you'll get 404s on all controller routes even though the DbContext is registered correctly. Web API projects (controllers, no views) use AddControllers(); Razor Pages projects use AddRazorPages(); Minimal API projects use neither — they map endpoints directly with app.MapGet() etc. Only AddControllersWithViews() enables both controller routing and .cshtml view rendering.
EF reads entity classes → builds in-memory model graph
② Model
🗺️ Model Building — entities, relationships, constraints conventions → annotations → Fluent API · each layer wins over the last
1
EF discovers entities via DbSet properties and reachable navigation properties, then applies conventions
Conventions handle the common cases automatically: Id or {ClassName}Id (e.g. UserId) → primary key; with NRT enabled (default in .NET 6+) string → NOT NULL nvarchar(max) and string? → nullable nvarchar(max); navigation properties → foreign keys inferred from naming.
2
Data Annotations override conventions inline on the entity class
Standard annotations (System.ComponentModel.DataAnnotations): [Required], [MaxLength], [Column], [Table]. EF Core-specific annotations (Microsoft.EntityFrameworkCore): [Index], [Precision], [Unicode]. Convenient for simple constraints but mixes persistence concerns into the domain model — prefer Fluent API for anything non-trivial.
3
Fluent API in OnModelCreating has the final word — overrides both conventions and annotations
Priority: Fluent API > Annotations > Conventions. Preferred for composite keys, owned entities (OwnsOne / OwnsMany), table splitting, TPH/TPT inheritance, and anything where you want persistence config kept out of the domain class. Use IEntityTypeConfiguration<T> to split large models into one file per entity.
4
Global query filters — declared via HasQueryFilter() in OnModelCreating, silently appended to every LINQ query for that entity type
The right place for multi-tenancy (WHERE TenantId = @current) and soft deletes (WHERE IsDeleted = 0). Override per-query with .IgnoreQueryFilters(). Note: filters do not apply to raw SQL or hand-written JOINs outside EF.
5
On first DbContext instantiation, EF compiles the completed model into an immutable cache — this is the design-time/runtime boundary
Subsequent instantiations reuse the cached model at near-zero cost. On large models this first-instantiation cost is measurable — use UseModel() with a pre-compiled static model to eliminate it entirely.
// ── Entity classes — conventions + annotations ──────────────────────────────
public class User
{
    public int    Id        { get; set; }                    // convention → PK (Id or UserId)
    [Required, MaxLength(100)]
    public string Name      { get; set; } = null!;           // NOT NULL nvarchar(100)
    public bool   IsDeleted { get; set; }                    // used by global query filter (step 4)
    public List<Post> Posts { get; set; } = [];            // nav property → EF infers FK on Post
}

public class Post
{
    public int    Id       { get; set; }
    public string Title    { get; set; } = null!;
    public int    AuthorId { get; set; }                    // convention → FK to User
    public User   Author   { get; set; } = null!;           // nav property back to User
}

// ── Fluent API — in AppDbContext.OnModelCreating ─────────────────────────────
protected override void OnModelCreating(ModelBuilder mb)
{
    mb.Entity<User>(b =>
    {
        b.ToTable("users");
        b.HasIndex(u => u.Name).IsUnique();
        b.HasQueryFilter(u => !u.IsDeleted);        // step 4 — appended to every User query
        b.HasMany(u => u.Posts)
         .WithOne(p => p.Author)
         .HasForeignKey(p => p.AuthorId)
         .OnDelete(DeleteBehavior.Cascade);
    });
}
💡 Large models: instead of growing OnModelCreating, implement IEntityTypeConfiguration<T> — one class per entity in its own file — then register everything with a single call: mb.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly).
dotnet ef migrations add → diffs model against snapshot
③ Migrations
🔄 Migrations — version control for your database schema scaffold · add · update · remove · script · bundle
What a migration actually is: EF diffs your current model against AppDbContextModelSnapshot.cs — a C# representation of the last known schema state — and generates the delta as a C# file with Up() and Down() methods. The generated files (<timestamp>_<Name>.cs, <timestamp>_<Name>.Designer.cs, and the updated snapshot) all go into source control. The database is never read during generation.

Prerequisite — install the EF CLI tool once per machine: dotnet tool install --global dotnet-ef. Verify with dotnet ef --version. If you get "dotnet: unknown command 'ef'" this is why. Pin the version to match your EF Core package: dotnet tool install --global dotnet-ef --version 8.0.10 (replace with your exact EF Core version).
Two workflows — choose one per project:

Code-first (steps 1–4 below) — you define C# classes, EF generates migrations, migrations drive the schema. The source of truth is your code.

Database-first / scaffold (step 0 below) — the database already exists; EF reverse-engineers it into C# classes and a DbContext. The source of truth is the database. You can re-scaffold at any time to pick up schema changes.
Database-first — reverse-engineer an existing schema
0
Database-first: dotnet ef dbcontext scaffold — reverse-engineer an existing database into entity classes and a DbContext
Reads the live database schema and generates one C# class per table plus a DbContext with all mappings pre-configured. Use -o to set the output folder, -c to name the context, --data-annotations to prefer attributes over fluent API, and --no-onconfiguring (EF Core 7 and below) / --no-OnConfiguring (EF Core 8+) to skip writing the connection string into the context — recommended, keep connection strings out of source control. Re-run with --force to overwrite when the schema changes. Tip: after the initial scaffold, consider switching to code-first for ongoing changes — the scaffold gives you the starting point, migrations take it forward.
# ── Database-first scaffold — reverse-engineer an existing SQL Server database ──
# Run from the project directory. Each flag explained below.
# Read the connection string from an env var — never hardcode in scripts.
# Set it first:  export DB_CONN="Server=(localdb)\MSSQLLocalDB;Database=OrderDb;Trusted_Connection=True;TrustServerCertificate=True;"

dotnet ef dbcontext scaffold \
    "$DB_CONN" \
    Microsoft.EntityFrameworkCore.SqlServer \
    -o Models \
    -c OrderDbContext \
    --no-onconfiguring \
    --data-annotations

# -o Models               output folder for generated entity classes
# -c OrderDbContext       name of the generated DbContext class
# --no-onconfiguring      don't embed the connection string in OnConfiguring()
#                         EF Core 8+ renamed this: --no-OnConfiguring (capital O and C)
# --data-annotations      use [Key],[Required] etc. instead of pure fluent API
#
# Re-scaffold after schema changes — overwrites generated files:
# dotnet ef dbcontext scaffold "$DB_CONN" ... --force
# Keep custom logic in partial classes so --force doesn't wipe it.

Code-first — define models in C#, let EF own the schema
1
dotnet ef migrations add <Name> — generate a migration from the current model diff; edit Up() and Down() freely before applying
Creates <timestamp>_<Name>.cs (the migration), <timestamp>_<Name>.Designer.cs (EF metadata), and updates AppDbContextModelSnapshot.cs (the new baseline). Name should describe the change — AddUserNameIndex, CreateOrdersTable.
2
dotnet ef database update — apply pending migrations to the configured database
EF checks __EFMigrationsHistory first and skips already-applied migrations. Safe to run repeatedly. For local development, roll back to a specific migration with dotnet ef database update <MigrationName> — this calls Down() on everything after it. Do not use this as a production rollback strategy.
3
dotnet ef migrations remove — delete the last migration if it hasn't been applied yet
Safe escape hatch when you generated a migration and then changed your mind. Fails if the migration has already been applied — use dotnet ef database update <PreviousMigration> first to roll back, then remove.
4
Choose a deployment strategy — how migrations reach production matters as much as writing them
MigrateAsync() at startup — simple, good for single-instance containers; dangerous with multiple instances racing on boot. Mitigate with a distributed lock (IDistributedLock via Medallion.Threading or similar) or run migrations as a separate Kubernetes init-container / ECS task before the app starts. SQL script (dotnet ef migrations script) — generate idempotent SQL, review it, hand to a DBA or run in a pipeline; safest for production. Migration bundle (dotnet ef migrations bundle) — self-contained executable, recommended for CI/CD pipelines; runs migrations without the full app or EF tooling installed.
// ── Generated migration — edit Up() and Down() freely before applying ───────
public partial class AddUserNameIndex : Migration
{
    protected override void Up(MigrationBuilder mb)
    {
        mb.CreateIndex(name: "IX_users_Name", table: "users", column: "Name", unique: true);
    }

    protected override void Down(MigrationBuilder mb)
    {
        mb.DropIndex(name: "IX_users_Name", table: "users");
    }
}
// ── Deployment option 1: MigrateAsync() at startup ──────────────────────────
// Simple for containers; guard against race conditions in multi-instance deploys
await using var scope = app.Services.CreateAsyncScope();
await scope.ServiceProvider
           .GetRequiredService<AppDbContext>()
           .Database.MigrateAsync();
# ── Deployment option 2: SQL script — safest for production ─────────────────
# --idempotent checks __EFMigrationsHistory before each step — safe to re-run
dotnet ef migrations script --idempotent --output migrations.sql

# ── Deployment option 3: migration bundle — recommended for CI/CD ───────────
# Self-contained executable — no app or EF tooling needed at deploy time
dotnet ef migrations bundle --self-contained -r linux-x64
⚠️ Never edit a migration already applied to production. Once __EFMigrationsHistory records it, EF considers it applied permanently. Create a new migration to fix or revert the change.
⚠️ Down() is not a production rollback strategy. It's a local development convenience. In production, roll forward with a new corrective migration — attempting to run Down() on a live database with real data is dangerous and often destructive.
⚡ Runtime — runs per operation in your application
④ Query
🔍 Query Pipeline — LINQ to SQL expression tree → translated → cached → executed
IQueryable<T> is a description, not a result. Every .Where(), .OrderBy(), and .Select() builds an expression tree in memory — nothing touches the database. SQL is sent only when you enumerate: ToListAsync(), FirstOrDefaultAsync(), SingleOrDefaultAsync(), CountAsync(), or foreach. Every pitfall in this section — client evaluation, full-table loads, unnecessary tracking — traces back to misunderstanding this boundary.
1
EF translates the expression tree to SQL and caches the translation by query shape — first call compiles, subsequent calls reuse
Query shape is determined by the structure of the expression tree, not the parameter values — parameters are always sent separately, never interpolated into SQL. If EF cannot translate an expression, it throws at runtime with a clear message indicating what could not be translated. Keep logic that can't translate to SQL out of IQueryable chains.
2
Never break the IQueryable chain early — calling .AsEnumerable() or returning IEnumerable<T> mid-chain sends the SQL immediately and pulls all rows into memory
Everything chained after the break runs in C# against the already-loaded dataset. A common version of this: a repository method returns IEnumerable<T> instead of IQueryable<T>, so every caller's .Where() filters in memory on the full table.
3
Always project with Select() — load full entities only when you intend to modify them
A full entity fetch loads every column, allocates a tracked instance, and stores a snapshot copy. If you only need two fields for a read, project to an anonymous type or DTO. This is not a micro-optimisation — on wide tables or large result sets it is the difference between milliseconds and seconds.
4
Results are materialised and registered with the Change Tracker — use AsNoTracking() for read-only queries
Two costs apply to tracked queries: snapshot allocation happens at query time (one copy of every property value per entity); change detection (DetectChanges) runs inside SaveChanges to diff current vs snapshot values. AsNoTracking() skips both. See ⑤ for the full Change Tracker picture.
// IQueryable — built up, SQL not sent yet:
IQueryable<User> q = db.Users
    .Where(u => u.IsActive)
    .OrderBy(u => u.Name);            // still no SQL

List<User> users = await q.ToListAsync();  // ← SQL fires here

// IEnumerable pitfall — ALL rows loaded, filtering in C#:
IEnumerable<User> allUsersInMemory = db.Users
    .AsEnumerable()                   // ← SQL fires here, entire table loaded
    .Where(u => u.IsActive);           // filtered in memory — too late

// Projection — fetch only what you need:
var dtos = await db.Users
    .Where(u => u.IsActive)
    .Select(u => new { u.Id, u.Name })  // → SELECT Id, Name FROM users WHERE IsActive=1
    .ToListAsync();                    // projecting to non-entity type — no tracking, no snapshot, no unused columns

// Debug: see the SQL before it hits the database:
Console.WriteLine(q.ToQueryString());  // includes global query filters — useful for debugging silently-filtered results
⚠️ Global query filters are invisible. If a query returns fewer rows than expected, check whether a filter declared in OnModelCreating is excluding them. Use .IgnoreQueryFilters() to bypass, or ToQueryString() to see the full SQL including filter clauses.
Returned entities registered with the Change Tracker
⑤ Change Tracker
📋 Change Tracker — the per-scope state register snapshot · state machine · identity map · DetectChanges
The Change Tracker is what makes EF feel like magic. When you write user.Name = "Alice"; await db.SaveChangesAsync() with no explicit UPDATE statement, the tracker is doing the work — it noticed the property changed by diffing against its stored snapshot, marked the entity Modified, and translated that to SQL. Every tracked entity has a state, an original-values snapshot, and an entry in the identity map.
1
Every entity loaded by a tracked query is registered with state Unchanged and a snapshot of its original values
The snapshot is a deep copy of all scalar property values at load time. It's the baseline DetectChanges diffs against. You can inspect all tracked entries via db.ChangeTracker.Entries() — useful for debugging unexpected saves or missing updates.
2
Five entity states — transitions happen via property assignment, db.Add, db.Remove, db.Attach, or db.Entry(e).State
Detached — not known to the tracker (never loaded, or loaded with AsNoTracking)
Unchanged — loaded and tracked; no properties changed since snapshot
Added — new entity; will become an INSERT on SaveChanges
Modified — at least one property differs from the snapshot; will become an UPDATE
Deleted — marked for removal; will become a DELETE on SaveChanges
3
DetectChanges — the mechanism that scans all tracked entities, diffs current vs snapshot, and updates states
Runs automatically before SaveChanges, and is also triggered by ChangeTracker.Entries() and certain other operations. Expensive at scale — if you're loading thousands of entities and not saving them, disable auto-detection with db.ChangeTracker.AutoDetectChangesEnabled = false and call db.ChangeTracker.DetectChanges() manually when needed.
4
The Change Tracker also acts as an identity map — loading the same entity twice in one scope returns the same instance, not two separate copies
FindAsync(5) called twice hits the tracker on the second call and returns immediately without a database round-trip. This is the first-level cache. It means within one scope, your in-memory entity and the database row are guaranteed to be in sync — until someone else modifies the database.
5
AsNoTracking() — skip registration entirely; AsNoTrackingWithIdentityResolution() — skip snapshots but still de-duplicate related entities
AsNoTracking: results are Detached, no snapshot, no identity map entry — use for pure reads. AsNoTrackingWithIdentityResolution: useful when a query returns the same parent entity multiple times via Includes — de-duplicates in memory without the full tracking overhead. Neither can be used with SaveChanges.
// Tracked — EF watches for changes via snapshot diff:
var user = await db.Users.FindAsync(5);    // state: Unchanged — snapshot stored
user!.Name = "Alice";                       // state: Modified  — detected by DetectChanges
await db.SaveChangesAsync();                // → UPDATE users SET Name='Alice' WHERE Id=5

// Identity map — second FindAsync returns same instance, no DB call:
var a = await db.Users.FindAsync(5);       // hits database
var b = await db.Users.FindAsync(5);       // hits tracker — same instance, no round-trip
Console.WriteLine(object.ReferenceEquals(a, b)); // True

// AsNoTracking — read-only, no snapshot, no identity map:
var users = await db.Users
    .AsNoTracking()
    .Where(u => u.IsActive)
    .ToListAsync();                          // Detached — SaveChanges ignores these entirely

// Inspect what the tracker currently holds:
foreach (var entry in db.ChangeTracker.Entries())
    Console.WriteLine($"{entry.Entity.GetType().Name} — {entry.State}");
⚠️ Tracking has a memory cost. EF allocates a snapshot copy of every scalar property for every tracked entity. Loading 10,000 entities tracked roughly doubles memory usage. Use AsNoTracking() for lists, reports, and anything you won't modify.
SaveChanges called → DetectChanges → commands batched → SQL sent
⑥ SaveChanges
💾 SaveChanges — flush to the database detect → batch → transaction → execute
What happens inside SaveChangesAsync()
DetectChanges — compares all tracked entities against their snapshots, updates entity states
SavingChanges event / ISaveChangesInterceptor.SavingChangesAsync — hook fires before any SQL is sent
Command batch built — INSERTs, UPDATEs, DELETEs ordered to respect FK constraints
Transaction opened — implicit transaction wraps the entire batch (unless you supplied one)
SQL executed — commands sent to the database; concurrency tokens checked (see ⑧)
Transaction committed — all commands succeeded; changes are durable
Snapshots updated — tracked entities' original-value snapshots refreshed to the saved values
SavedChanges event / ISaveChangesInterceptor.SavedChangesAsync — hook fires after commit; safe to dispatch domain events here
● EF internal  ·  ● database operation  ·  ● interceptor / event hook 
// All three change states in one SaveChanges call:
var existingUser = await db.Users.FindAsync(3);   // state: Unchanged
var oldUser      = await db.Users.FindAsync(7);   // state: Unchanged

db.Add(new User { Name = "Bob" });               // state: Added   → INSERT
existingUser!.Name = "Alice";                    // state: Modified → UPDATE
db.Remove(oldUser!);                             // state: Deleted  → DELETE

int rowsAffected = await db.SaveChangesAsync();   // all three sent in one transaction

// ── SavingChanges hook ─────────────────────────────────────────────────────
// Subscribe ONCE in AppDbContext's constructor, not in a service method.
// Attaching inline (db.SavingChanges += ...) per-call leaks handlers — each
// call appends a new delegate that is never removed for the lifetime of the instance.
//
// Add to AppDbContext constructor:
public AppDbContext(DbContextOptions<AppDbContext> options) : base(options)
{
    SavingChanges += (_, _) => StampAuditFields();
}

// IAuditable — define once in your project, implement on any entity that needs audit stamps:
public interface IAuditable
{
    DateTime CreatedAt { get; set; }
    DateTime UpdatedAt { get; set; }
}

private void StampAuditFields()
{
    var now = DateTime.UtcNow;
    foreach (var entry in ChangeTracker.Entries<IAuditable>())
    {
        if (entry.State == EntityState.Added)
        {
            entry.Entity.CreatedAt = now;   // set on INSERT only
            entry.Entity.UpdatedAt = now;
        }
        else if (entry.State == EntityState.Modified)
        {
            entry.Entity.UpdatedAt = now;   // leave CreatedAt untouched
        }
    }
}
💡 Unit of Work pattern: DbContext is the Unit of Work. Make all your changes, then call SaveChanges once. Don't call it after every individual operation — the whole point is batching related changes into one atomic transaction.
Navigation properties resolved — JOIN vs separate query
⑦ Relationships
🔗 Relationships & Loading Strategies eager · explicit · lazy — pick one
1
Eager loading — Include() / ThenInclude() — loads related data in the same query via JOIN
Best when you know you'll always need the related data. One round-trip to the database. Watch out for cartesian explosion: including two collection navigations (e.g. Posts and Tags) generates a JOIN that multiplies rows — 100 users × 10 posts × 5 tags = 5,000 rows. Use AsSplitQuery() to break it into separate SELECTs (see ⑫ Performance). For single collection includes, a JOIN is usually cheaper than the extra round-trip.
2
Explicit loading — entry.Collection().LoadAsync() — you ask for related data on demand
Good when related data is only needed conditionally. Use entry.Collection(u => u.Posts).LoadAsync() for one-to-many, and entry.Reference(p => p.Author).LoadAsync() for a single reference. Before loading, check entry.Collection(u => u.Posts).IsLoaded to avoid a redundant round-trip if it was already fetched by a prior query in the same scope.
3
Lazy loading — navigation property access automatically fires a SQL query
Requires Microsoft.EntityFrameworkCore.Proxies and UseLazyLoadingProxies(), plus virtual nav properties. Convenient but hides N+1 bugs: in a loop over 100 users, accessing user.Posts each iteration fires 100 separate queries — 101 total instead of 1. This often goes unnoticed until production load reveals it. Recommendation: disable lazy loading by default; use Include() or explicit loading instead, so data access is visible in code.
// Eager — one JOIN query (2 includes = one wide SELECT with two JOINs):
var usersWithPosts = await db.Users
    .Include(u => u.Posts)
        .ThenInclude(p => p.Comments)
    .ToListAsync();

// Split query — avoids cartesian explosion for large collections:
var usersSplit = await db.Users
    .Include(u => u.Posts)
    .AsSplitQuery()    // → 2 separate SELECTs instead of one wide JOIN
    .ToListAsync();

// Explicit — load only when needed:
var user = await db.Users.FindAsync(5);   // returns User? — null if not found
if (user is not null)
    await db.Entry(user).Collection(u => u.Posts).LoadAsync();
⚠️ The N+1 problem: iterating a list and accessing a navigation property inside the loop fires one SQL query per entity. 100 users → 101 queries. Always Include() upfront or project with Select() if you need related data inside a loop.
Concurrent updates detected — concurrency token checked
⑧ Concurrency
⚡ Concurrency — detecting conflicting writes optimistic by default · rowversion recommended
1
Mark a property with [ConcurrencyToken] or use a rowversion column — EF adds it to every UPDATE's WHERE clause
EF checks that the token hasn't changed since you loaded the entity. If it has, someone else modified the row — the UPDATE affects 0 rows. SQL Server rowversion / timestamp (mapped as byte[] with [Timestamp]): auto-incremented by the database on every write, requires no application code, and is the recommended approach. Manual token (e.g. Guid or DateTime with [ConcurrencyToken]): you update it yourself before saving — easier to port across databases but relies on application discipline to keep it current.
2
When 0 rows are affected, EF throws DbUpdateConcurrencyException — you must catch and resolve it
The exception carries three value sets: entry.OriginalValues (what you loaded), entry.CurrentValues (what you tried to save), and await entry.GetDatabaseValuesAsync() (what's in the database right now). These give you everything needed to implement any resolution strategy — see the code block below for all three approaches.
public class User
{
    public int     Id         { get; set; }
    public string? Name       { get; set; }
    [Timestamp]                               // SQL Server rowversion — auto-increments on every write
    public byte[]? RowVersion { get; set; }  // nullable — EF populates after INSERT/load
                                               // → WHERE Id=5 AND RowVersion=0x0000000000000042
}

// ── Strategy 1: Client wins — retry with your values, ignoring what's in the DB ──
try { await db.SaveChangesAsync(); }
catch (DbUpdateConcurrencyException ex)
{
    var entry    = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();
    entry.OriginalValues.SetValues(dbValues!); // bump token to DB's current value
    await db.SaveChangesAsync();               // retry — your CurrentValues overwrite the DB
}

// ── Strategy 2: Server wins — discard your changes, keep what's in the DB ────
try { await db.SaveChangesAsync(); }
catch (DbUpdateConcurrencyException ex)
{
    var entry    = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();
    entry.SetValues(dbValues!);               // overwrite your proposed values with DB values
    entry.OriginalValues.SetValues(dbValues!); // sync token — no SaveChanges needed
}

// ── Strategy 3: Merge — decide per-property (e.g. surface a conflict UI) ─────
try { await db.SaveChangesAsync(); }
catch (DbUpdateConcurrencyException ex)
{
    var entry    = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync();
    var current  = entry.CurrentValues;
    foreach (var prop in entry.Metadata.GetProperties())
    {
        // MergePolicy is application-specific — pick yours, client, server or combined:
        // e.g. take DB value for Price, keep client value for Description
        current[prop] = current[prop] ?? dbValues![prop]; // example: keep client if non-null
    }
    entry.OriginalValues.SetValues(dbValues!); // sync token before retry
    await db.SaveChangesAsync();               // save the merged result
}
Bypass the pipeline — raw SQL or stored procedures
⑨ Raw SQL
🛠️ Raw SQL & Stored Procedures — when LINQ isn't enough composable or non-composable · watch for injection
1
FromSql() — returns entity instances that compose with LINQ and are tracked
The SQL becomes a subquery — you can add .Where(), .OrderBy(), .Include() on top. Must return all columns the entity expects. FromSql($"…{variable}…") (interpolated): EF converts interpolated values to parameterised placeholders — safe against injection. FromSqlRaw("…" + variable): raw string concatenation — injection risk if any part comes from user input. Always prefer FromSql with interpolation; use FromSqlRaw only when the SQL itself is a static constant with no user input.
2
ExecuteUpdateAsync / ExecuteDeleteAsync (.NET 7+) — LINQ-based bulk DML that bypasses the Change Tracker entirely
One SQL statement, no entities loaded, no snapshots. For raw DML strings or stored proc calls, use ExecuteSqlRaw() / ExecuteSqlAsync() instead — same bypass, but takes a hand-written SQL string. (ExecuteSqlInterpolated still compiles but is obsolete as of EF Core 7 — use ExecuteSqlAsync with interpolation instead.)
3
SqlQuery<T>() (.NET 8+) — raw SQL that returns non-entity types (e.g. scalar projections, DTOs)
No longer requires a DbSet or keyless entity type — project to any record, class, or primitive. The result is not tracked. Use when the shape of what you want back doesn't match any of your mapped entities — e.g. a report that joins across three tables and returns a flattened summary row.
// FromSql — composable, tracked, parameterised:
var users = await db.Users
    .FromSql($"SELECT * FROM users WHERE TenantId = {tenantId}")  // interpolation → safe parameter
    .Where(u => u.IsActive)   // composed as SQL subquery
    .ToListAsync();

// ExecuteUpdate — bulk UPDATE without loading entities (.NET 7+):
await db.Users
    .Where(u => u.LastLogin < DateTime.UtcNow.AddYears(-1))
    .ExecuteUpdateAsync(s => s.SetProperty(u => u.IsActive, false));
// → UPDATE users SET IsActive=0 WHERE LastLogin < @p0   (no entities loaded)

// SqlQuery — raw SQL returning a non-entity type (.NET 8+):
record OrderSummary(int OrderId, string CustomerName, decimal Total);

var summaries = await db.Database
    .SqlQuery<OrderSummary>($"""
        SELECT o.Id AS OrderId, c.Name AS CustomerName, SUM(li.Price) AS Total
        FROM Orders o
        JOIN Customers c ON c.Id = o.CustomerId
        JOIN LineItems li ON li.OrderId = o.Id
        WHERE o.TenantId = {tenantId}
        GROUP BY o.Id, c.Name
    """)
    .ToListAsync();
// Returns List<OrderSummary> — not tracked, no DbSet required

// ExecuteSqlAsync — call a stored procedure (safe interpolation, EF Core 7+):
await db.Database.ExecuteSqlAsync(
    $"EXEC sp_ArchiveOrders @cutoff={cutoffDate}, @batchSize={batchSize}");
// interpolated values become SQL parameters — safe against injection
⚠️ Always use parameterised queries. FromSql($"…{variable}…") (interpolated) is safe — EF converts interpolated values to parameters. FromSqlRaw("…" + variable) is string concatenation and is vulnerable to SQL injection. Prefer the interpolated form.
Cross-cutting hooks — intercept commands and save operations
⑩ Interceptors
🔌 Interceptors & Events — cross-cutting hooks command · save · connection · transaction
1
IDbCommandInterceptor — intercept every SQL command before and after execution
Use for query logging, slow-query alerts, read/write splitting. Override ReaderExecutingAsync to inspect/modify a query before it runs; override ReaderExecutedAsync to log duration after. For lightweight logging without a full interceptor, use optionsBuilder.LogTo(Console.WriteLine, LogLevel.Information) or wire up a DiagnosticSource listener — DiagnosticListener("Microsoft.EntityFrameworkCore") emits structured events for every command, connection, and transaction without subclassing.
2
ISaveChangesInterceptor — hook into SaveChanges before and after the transaction commits
Best place for audit logging, soft-delete enforcement, domain event dispatch. SavingChangesAsync: fires before SQL is sent — use to stamp audit fields, enforce invariants, or enqueue outbox messages as part of the same transaction. SavedChangesAsync: fires after the transaction commits — use to dispatch domain events, invalidate caches, or trigger side-effects that should only happen if the save succeeded. Never dispatch events in SavingChangesAsync; the transaction may still roll back.
3
IConnectionInterceptor / ITransactionInterceptor — hook into connection open/close and transaction begin/commit/rollback
Useful for read/write connection splitting (send SELECTs to a read replica, writes to primary) and distributed tracing. Note: global query filters (multi-tenancy, soft deletes) belong in OnModelCreating — see ② Model.
public class SlowQueryInterceptor : DbCommandInterceptor
{
    private readonly ILogger<SlowQueryInterceptor> _log;

    public SlowQueryInterceptor(ILogger<SlowQueryInterceptor> log) => _log = log;

    public override ValueTask<DbDataReader> ReaderExecutedAsync(
        DbCommand command, CommandExecutedEventData eventData, DbDataReader result,
        CancellationToken ct = default)
    {
        if (eventData.Duration > TimeSpan.FromMilliseconds(500))
            _log.LogWarning("Slow query ({Ms}ms): {Sql}",
                (long)eventData.Duration.TotalMilliseconds, command.CommandText);
        return new ValueTask<DbDataReader>(result);
    }
}

// When an interceptor takes constructor-injected services, register it separately
// and resolve via 'sp' — simple interceptors with no dependencies can use 'new' directly:
builder.Services.AddSingleton<SlowQueryInterceptor>();
builder.Services.AddDbContext<AppDbContext>((sp, o) =>
    o.UseSqlServer(connStr)
     .AddInterceptors(sp.GetRequiredService<SlowQueryInterceptor>()));
ADO.NET manages the actual connection pool
⑪ Connections
🗄️ Connection & Transaction Management pooled by ADO.NET · explicit transactions available
1
Connection pooling is handled by ADO.NET (SqlClient), not EF — EF borrows and returns connections from the pool
EF opens a connection only when it needs to execute SQL and closes it immediately after. The pool keeps the physical TCP connection alive. Key connection string parameters: Max Pool Size (default 100 — max simultaneous connections; raise for high-throughput APIs, don't lower without reason), Min Pool Size (default 0 — connections kept warm; set to a small value on servers with constant load to avoid cold-start latency), Connect Timeout (default 15s — how long to wait for a pool slot or new connection before throwing). Monitor pool exhaustion: if all connections are in use, new requests queue until timeout — a sign you need a larger pool or shorter-lived DbContexts.
2
Implicit transactions — SaveChanges wraps its batch in a transaction automatically
You get atomicity for free. If any command fails, the whole batch rolls back. You don't need to write transaction code for normal save operations.
3
Explicit transactions — BeginTransactionAsync() — span multiple SaveChanges calls or raw SQL
BeginTransactionAsync() returns an IDbContextTransaction — call CommitAsync() to commit or let await using dispose it (which rolls back if commit was never called). To share a transaction across two DbContext instances, call db2.Database.UseTransactionAsync(tx.GetDbTransaction()) — both contexts then participate in the same ADO.NET transaction. Avoid TransactionScope with async EF code: it can silently promote to a distributed transaction (DTC) if EF opens more than one connection, which requires MSDTC, adds significant overhead, and is disabled by default in many cloud environments.
// Explicit transaction spanning two SaveChanges calls:
await using var tx = await db.Database.BeginTransactionAsync();
// 'await using' guarantees rollback on any unhandled path — CommitAsync() was never called.
try
{
    db.Add(new Order { /* properties omitted */ });
    await db.SaveChangesAsync();            // first batch

    db.Add(new Invoice { /* properties omitted */ });
    await db.SaveChangesAsync();            // second batch — same transaction

    await tx.CommitAsync();
}
catch
{
    // No RollbackAsync() needed — 'await using' disposes the transaction automatically,
    // rolling back if CommitAsync() was never reached.
    throw;
}
Tune for production — compiled queries, bulk ops, projections
⑫ Performance
🚀 Performance Pitfalls & Optimisations compiled queries · bulk ops · projections · split queries · pre-compiled model
1
Compiled queries — pre-translate a fixed LINQ query shape, skip the expression-tree compilation on every call
Biggest win for hot paths (login checks, per-request lookups). Use EF.CompileAsyncQuery() to produce a static delegate — the expression tree is compiled to SQL once and the result cached in a static field. The query shape must be fixed at compile time; you can pass parameters but not dynamically build the query structure. Not worth it for queries called rarely or already cached at a higher layer.
2
Project, don't load — Select() to a DTO fetches only the columns you need
Loading full tracked entities when you only need two fields wastes memory, bandwidth, and Change Tracker overhead. Rule: never load what you won't use.
3
Bulk operations — ExecuteUpdateAsync / ExecuteDeleteAsync (.NET 7+) bypass the Change Tracker entirely
One SQL statement, no entities loaded, no snapshots, no DetectChanges. Orders of magnitude faster for bulk writes. Contrast: updating 10,000 rows the traditional way (load all → modify each → SaveChanges) allocates 10,000 tracked entities with 10,000 snapshots, runs DetectChanges over all of them, then sends 10,000 individual UPDATE statements (or a large batch). ExecuteUpdateAsync sends a single UPDATE … WHERE … regardless of row count. Trade-off: bypasses interceptors and does not fire domain events.
4
Split queries — AsSplitQuery() avoids cartesian explosion when including multiple collection navigations
A JOIN across two collection includes multiplies rows: 100 users × 10 posts × 5 comments = 5,000 rows returned. Split into 3 separate queries instead. Trade-off: single JOIN query — one round-trip, consistent point-in-time snapshot, but can produce massive result sets; better when collections are small or you include only one collection. Split query — smaller result sets, more predictable memory use, but multiple round-trips and a tiny risk of inconsistency if data changes between the queries. In practice, split queries win once any included collection exceeds ~50 rows.
5
Log your queries — enable EF logging or use ToQueryString() to see the SQL before it hits production
Three ways to see your SQL: ToQueryString() on any IQueryable returns the SQL as a string inline — good for one-off debugging. optionsBuilder.LogTo(Console.WriteLine, LogLevel.Information) logs every command to the console — add it in the AddDbContext lambda for DI apps, or in OnConfiguring for non-DI tools. EnableSensitiveDataLogging() adds parameter values to the log output (never enable in production). For production, use MiniProfiler or a structured logging interceptor to surface slow queries without the noise of logging all SQL.
6
Pre-compiled model — eliminate the one-time model compilation cost on first DbContext instantiation
Run dotnet ef dbcontext optimize to generate a static compiled model, then wire it up via UseModel(). Worth doing when startup time is measurable — large models with hundreds of entities can add hundreds of milliseconds to cold start.
// Compiled query — translate once, reuse forever:
// CompileAsyncQuery takes a synchronous LINQ expression and bakes the async Task<T> wrapper
// into the delegate — that's why the lambda uses FirstOrDefault (not FirstOrDefaultAsync)
// yet the Func return type is Task<User?> and the call site uses await.
private static readonly Func<AppDbContext, int, Task<User?>> GetUserById =
    EF.CompileAsyncQuery((AppDbContext db, int id) =>
        db.Users.FirstOrDefault(u => u.Id == id));

var user = await GetUserById(db, 5);   // no expression-tree compilation cost

// See the SQL before you run it:
// ToQueryString() reflects runtime state at the moment of the call — global filters,
// query filters, and any scoped parameters (e.g. current tenant) are evaluated then.
// Calling it outside a normal request scope may produce different SQL than production.
var query = db.Users.Where(u => !u.IsDeleted).OrderBy(u => u.Name);
Console.WriteLine(query.ToQueryString());  // prints the SQL — global filter also applied

// Pre-compiled model — generated via: dotnet ef dbcontext optimize
// Namespace defaults to {RootNamespace}.CompiledModels; override with --namespace
builder.Services.AddDbContext<AppDbContext>(o =>
    o.UseSqlServer(connStr)
     .UseModel(CompiledModels.AppDbContextModel.Instance));
💡 Quick checklist before shipping a query: Is it using AsNoTracking() if read-only? Is it projecting with Select() rather than loading full entities? Does it have an Include() to avoid N+1? Does the WHERE clause translate to SQL (not client-eval)? Have you checked ToQueryString()?
